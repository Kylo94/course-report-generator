---

## 一、项目概述

### 1.1 一句话定义
为少儿编程教师设计的**课程报告自动生成工具**：选定学生本节课的程序文件夹，工具自动提取代码、调用 AI 生成报告内容、结合老师手动上传的运行截图，按 4 页 A4 模板输出一份排版精美的课程报告 PDF。

### 1.2 解决的核心问题
少儿编程教师每节课后需为每位学生撰写课程报告（课后反馈），内容涉及本节知识点、学习评价、课后作业等。当前痛点：

- **重复性高**：同一份课程内容需为 N 名学生各写一份
- **耗时大**：单份报告撰写 + 排版约 20-30 分钟
- **质量不稳**：教师精力有限时评价流于形式
- **家长沟通不连续**：缺乏结构化记录与历史回溯

### 1.3 价值主张
- 单份报告生成耗时从 **20-30 分钟** 降至 **3-5 分钟**
- 报告内容**结构统一、专业度提升**
- 学生学习轨迹**可追溯**（数据库沉淀历史课程）
- 老师角色从"写作者"转变为"审阅者"，**专注教学质量本身**

## 二、目标用户与场景

### 2.1 用户画像
- **职业**：少儿编程机构教师或自由职业少儿编程教师
- **技术能力**：能熟练运行 Python 环境，但非专业程序员
- **使用频率**：每节课后使用 1-2 次（一周 3-10 节课）
- **班级规模**：单次课程通常带 4-15 名学生

### 2.2 典型使用场景
```
课前：老师确定本节课教学主题（如"飞翔的小鸟第一课"）
课中：学生在老师指导下完成课堂项目
课后：
  Step 1：老师在工具中选择本节课学生的程序文件夹（批量）
  Step 2：点击"生成报告"，AI 一次性生成 9 项内容
  Step 3：老师在界面中微调措辞、补充个性化评价
  Step 4：上传本节课运行截图
  Step 5：选定排版模板，一键导出 PDF
  Step 6：发送给家长 / 归档
```

### 2.3 不在本期范围
- ❌ 多教师协同 / 账号系统
- ❌ 学员家长端 / 学员端
- ❌ 课程管理 SaaS 化
- ❌ 多语言支持
- ❌ 离线模式（必须联网）
- ❌ 付费授权 / 模板市场

## 三、需求分析

### 3.1 功能需求

#### F1. 项目文件夹管理
- 选择本地程序文件夹（含 Python 源码 + 资源）
- 自动识别启动文件（约定 `main.py` 或首层 `.py`）
- 解析启动文件顶部注释（格式 `# Course: <主题>`）
- 列出项目内所有 `.py` 文件及依赖资源

#### F2. AI 内容生成
基于项目代码 + 注释，**一键生成**以下 9 项内容：

| 序号 | 字段 | 说明 | 限制 |
|------|------|------|------|
| 1 | 上课时间 | 默认今日，可手动修改 | YYYY-MM-DD |
| 2 | 课程名称 | 优先从注释解析，否则 AI 推断 | ≤ 10 字 |
| 3 | 知识点概括 | 列表形式 | ≤ 5 条，每条 ≤ 15 字 |
| 4 | 能力提升 | 一句话 | 30-80 字 |
| 5 | 单词学习 | 英文单词 + 释义 + 例句 | 1 个 |
| 6 | 内容概述 | 逐知识点展开叙述 + 难点 | 60-100 字/条 |
| 7 | 截图 | 手动上传（≥1 张） | jpg/png/webp |
| 8 | 作业 | 含目标、提示、评分点 | 结构化输出 |
| 9 | 学生评价 | 口语化，6 维度 | 180-220 字 |

**生成策略**：3 步链式生成（知识点 → 概述/能力 → 作业/评价），每步可单独重试。

#### F3. 多学生差异化
- 一键生成 N 份报告时，**保证学生评价不同**：
  - 引入 `temperature=0.9` 提高多样性
  - 评价分 6 维度（专注力、逻辑理解、动手能力、协作、表达、创意），按学生特征随机组合
- 同节课共用知识点，差异化仅在"作业难度"和"学生评价"

#### F4. 学生管理
- 增删改查学生档案：姓名、年龄、性别、基础水平、性格特点、家长联系方式
- 班级管理：按班级批量选择学生
- 批量导入：Excel/CSV
- 学生卡片：历次课程报告快捷入口
- 数据导出：JSON 备份

#### F5. 报告排版与导出
- 内置 ≥ 3 套 A4 模板（经典简约 / 少儿卡通 / 学术风）
- 支持自定义排版（JSON 模板文件 + 可视化编辑器）
- 页面背景可设置（图片/纯色/渐变）
- 4 页 A4，图文混排
- 输出格式：PDF（主）、可选 PNG 单页导出

##### F5.1 Logo 配置
支持上传教师所属机构/工作室的 Logo 并灵活摆放，提升报告品牌识别度。

**基础能力：**
- 一次上传，全局生效：上传一张 Logo，应用于所有报告
- 支持格式：PNG（推荐，透明背景）/ JPG / SVG
- 推荐尺寸：方形 512×512 ~ 1024×1024
- 存储位置：`./data/assets/logo.{ext}`

**位置（6 种）：**

| 位置 | 适用场景 |
|------|---------|
| 左上（top-left） | 最常见，专业感 |
| 右上（top-right） | 配合左侧标题文本 |
| 居中顶部（top-center） | 机构 Logo 居中展示 |
| 左下（bottom-left） | 与签名/页脚同侧 |
| 右下（bottom-right） | 经典版式 |
| 居中底部（bottom-center） | 页面底部水印式 |

**尺寸（3 档，**保持原宽高比**，以"宽度上限"为锚点）：**

> **核心约束**：Logo **始终保持原始宽高比**，不拉伸、不裁剪、不变形。
> 渲染时仅指定"宽度上限"，高度由原图比例自动计算。

| 档位 | 宽度上限 | 实际高度 | 适用 |
|------|---------|---------|------|
| 小（small） | 20mm | 由原图比例决定 | 信息密度高的模板 |
| 中（medium，默认） | 30mm | 由原图比例决定 | 通用 |
| 大（large） | 45mm | 由原图比例决定 | Logo 主导（机构宣传型） |

**实现要点：**
- **PDF（WeasyPrint）**：CSS `width: <size>mm; height: auto;`
- **Word（python-docx）**：`doc.add_picture(path, width=Mm(<size>))`，高度自动
- **前端预览**：`<img :width="size + 'mm'">`，高度由 CSS 自动
- **极扁/极高图片**：仍按原比例缩放，但若纵横比 > 3:1 或 < 1:3，UI 提示"建议使用更接近方形的 Logo"

**显示范围：**
- 默认：4 页 A4 每页都显示
- 可选：仅首页（封面）显示
- 可选：仅末页（评价/签名页）显示

**模板级覆盖：**
- 模板 JSON 中可独立指定 `logo.image_path`，覆盖全局 Logo
- 适用于"不同课程类型用不同 Logo"的场景

**Word 导出兼容：**
- 同一 Logo 同时嵌入 PDF 与 docx，位置/尺寸保持一致
- python-docx 通过 `add_picture()` 插入并用 `Inches()` 控制大小
- docx 中 Logo 浮动在页眉或段落旁

#### F6. 报告中心
- 历史报告按学生/班级/日期检索
- 单份报告二次编辑（调 AI 措辞）
- 一键复制导出到本地文件夹

#### F7. 草稿保存与续编
报告数据**在排版完成后、PDF 输出前**即可保存为草稿，老师可随时重新打开继续修改。

**保存触发：**
- 自动保存：编辑停顿 30 秒后自动落库
- 手动保存：点击"保存草稿"按钮
- 强制保存：关闭编辑窗口 / 退出应用前

**草稿包含的完整状态：**
- 9 项内容（已生成或已手动编辑的版本）
- 选定的排版模板 ID
- 已上传的截图列表
- 关联的学生 ID、课程日期
- AI 生成的元信息（用于"重新生成"时复用上下文）

**续编入口：**
- 报告中心列表（状态标记：草稿 / 已导出 / 已归档）
- 点击草稿 → 自动加载到 ReportEditor
- 已导出的报告也可"重新打开编辑"

**状态机：**
```
[新建中] ──保存──→ [草稿] ──导出PDF──→ [已导出] ──归档──→ [已归档]
                        │                                    │
                        └──────重新打开编辑────────────────────┘
```

**数据存储：** 草稿与正式记录共用 `course_records` 表，通过 `status` 字段区分。

#### F8. Word 文档导入导出
允许老师将报告导出为 Word 文档进行二次编辑，再导入回工具重新生成 PDF。

##### 8.1 导出 Word

**技术选型：原生 python-docx 生成**（非 PDF 转 Word）

| 方案 | 优势 | 劣势 | 选择 |
|------|------|------|------|
| **python-docx 原生生成** | 真正可编辑、字段结构保留、文件小 | 需维护 Word 样式系统 | ✅ **采用** |
| PDF → Word（pdf2docx） | 与 PDF 版式一致 | 转有损，文本变图片，无法编辑 | ❌ |

**设计要点：**
- 每个字段使用 Word 内置样式（`Heading 1` / `Heading 2` / `Normal`）
- 字段名称作为段落标题（如"课程主题"、"知识点概括"）
- 截图按比例插入，文字环绕
- 表格用于知识点、作业（目标/提示/评分点）
- 内置 Word 样式与 PDF 模板共享同一份 `theme` 配置（颜色、字体）

**生成的 docx 结构示例：**
```
[标题] 课程报告
[Heading 1] 基本信息
  上课时间：2026-06-29
  学生姓名：张小明
  课程名称：飞翔的小鸟第一课
[Heading 1] 知识点概括
  • print 的使用
  • 赋值运算符
  • 分支语句
[Heading 1] 内容概述
  [图片]
  1. print 的使用：学生通过 print() 函数输出小鸟的位置和得分……
  2. ...
[Heading 1] 单词学习
  Character /ˈkærəktər/ n. 角色
  例句：bird = Character("bird.png")
[Heading 1] 课后作业
  目标：...
  提示：...
  评分点：...
[Heading 1] 学生评价
  张小明本节课专注力很强，能够跟随老师的节奏...
```

##### 8.2 导入 Word

**导入策略：基于 Word 样式 + 字段标记混合识别**

1. **解析 docx 文档结构**，识别 Heading 1 段落
2. 匹配字段名（如"课程主题"→ `course_topic`）
3. 对模糊匹配失败的段落，**调用 AI 辅助识别**（LLM 一次性读全文 → 返回字段映射 JSON）
4. 在前端预览"识别结果"，老师确认后填回编辑界面
5. 截图按图片位置顺序匹配回原 `screenshot_paths`

**降级策略：** AI 识别失败时，按段落顺序填入"自定义内容"字段，老师可手动归类。

| 层 | 选型 | 理由 |

| 打包 | PyInstaller | 单文件夹发布 |

### 4.2 LLM 供应商设计

> **重要决策**：LangChain 不内置特定 LLM 供应商，但**必须由用户在配置中设置**（API Key + 默认模型）。

#### 4.2.1 LangChain 使用边界（采用方案 A：组件 + 自定义编排）

本项目在以下层面使用 LangChain：

| 使用层 | LangChain 组件 | 用途 |
|--------|---------------|------|
| LLM 抽象 | `langchain_<provider>.Chat<Model>` | 屏蔽多家供应商差异 |
| Prompt 模板 | `ChatPromptTemplate` | 模板化 4 套 Prompt |
| 结构化输出 | `PydanticOutputParser` / `JsonOutputParser` | 解析 AI 返回的 JSON |
| 流式输出 | `.stream()` | WebSocket 推流到前端 |

**不使用的 LangChain 能力：**

- ❌ **不使用 `Chain` / `SequentialChain` / `LCEL` 串联**
- ❌ 不使用 LangChain Agent / Tool
- ❌ 不使用 LangChain Memory

**顶层编排由业务代码控制**，原因：

1. **单步可重试**：老师点"重新生成知识点"时，不应触发整个 Chain 重跑
2. **断点续跑**：每步中间结果落库，AI 失败可从断点继续
3. **跨域数据**：学生评价依赖"学生档案 + 知识点 + 老师观察"，跨多个数据源
4. **可调试性**：业务代码编排比 LCEL 嵌套链更易追踪

**代码示意：**
```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import JsonOutputParser
from langchain_openai import ChatOpenAI  # 供应商可换

llm = ChatOpenAI(model=config.default_model, temperature=0.3)

# 每个子能力是一个独立的 LangChain Runnable，可单独 invoke/stream
extract_kp_chain = extract_kp_prompt | llm | JsonOutputParser()
extract_content_chain = extract_content_prompt | llm | StrOutputParser()

# 顶层：业务代码编排
class AIOrchestrator:
    def generate_report(self, project, student) -> ReportData:
        kp = extract_kp_chain.invoke({"code": project.code, "topic": project.topic})
        content = extract_content_chain.invoke({"kp": kp, "project_type": project.type})
        homework = extract_homework_chain.invoke({"kp": kp, "level": student.base_level})
        evaluation = extract_eval_chain.invoke({
            "kp": kp, "characteristics": student.characteristics, "level": student.base_level
        })
        return ReportData(knowledge_points=kp, content=content, homework=homework, evaluation=evaluation)
    
    def regenerate_one_field(self, field_name: str, context: dict):
        """单步重试：仅重跑指定字段，其他字段保持不变"""
        chain = self._chains[field_name]
        return chain.invoke(context)
```


支持的供应商（用户配置其中之一即可）：

| 供应商 | 推荐模型 | 适用 |
|--------|---------|------|
| DeepSeek | deepseek-chat | 性价比首选，中文好 |
| 通义千问 | qwen-plus | 阿里云稳定 |
| 智谱 GLM | glm-4 | 教育领域 |
| OpenAI | gpt-4o-mini | 兜底方案 |
| Claude | claude-sonnet-4-6 | 高质量长文本 |

#### LLMProvider 接口
```python
from abc import ABC, abstractmethod
from typing import Iterator

class LLMProvider(ABC):
    name: str
    
    @abstractmethod
    def chat(self, messages: list[dict], **kwargs) -> str: ...
    
    @abstractmethod
    def stream_chat(self, messages: list[dict], **kwargs) -> Iterator[str]: ...
    
    @abstractmethod
    def count_tokens(self, text: str) -> int: ...
```

#### 配置入口
- 配置文件：`./config/llm.yaml`
```yaml
provider: deepseek  # 用户指定
api_key: sk-xxxxx
base_url: https://api.deepseek.com/v1
default_model: deepseek-chat
timeout: 60
max_retries: 2
```
- 前端提供"LLM 设置"页：填写 Key → 测试连通 → 选择默认模型

### 4.3 AI Prompt 模板设计

#### 模板 1：知识点提取
```text
你是少儿编程教学助手。请阅读以下 Python 教学项目的源代码与注释，
提取本节课涉及的知识点，输出 JSON 列表（≤ 5 条，每条 ≤ 15 字）。

代码：
{code_content}

注释（课程主题）：{course_topic}

输出格式：
["知识点1", "知识点2", ...]
```

#### 模板 2：内容概述 + 能力提升
```text
你是少儿编程教学助手。基于以下知识点，为家长写一段课程概述，
每条 60-100 字，说明如何使用 + 难点在哪。
再总结 30-80 字"能力提升"。

知识点：{knowledge_points}
项目类型：{project_type}
```

#### 模板 3：作业生成
```text
你是少儿编程老师。为本节课布置一份难度不高的课后作业，要求：
1. 巩固本节知识点
2. 包含 目标、提示、评分点
3. 学生基础水平：{student_level}

知识点：{knowledge_points}
```

#### 模板 4：学生评价（差异化关键）
```text
你是少儿编程老师。基于以下信息，为该学生写 180-220 字口语化评价。

学生特点：{characteristics}
学生基础：{base_level}
本节知识点：{knowledge_points}
本节课堂表现观察：{teacher_observation}

评价维度（覆盖即可，无需都写）：
- 专注力
- 逻辑理解  
- 动手能力
- 协作意识
- 表达能力
- 创意性

要求：避免重复"上课认真""积极举手"等套话，给出具体场景。
```

### 4.4 报告生成流程

```
[选择项目文件夹]
       │
       ▼
[静态分析: AST + 注释解析] ──→ 课程主题、代码摘要
       │
       ▼
[AI 链式生成 - LangChain 组件 + 自定义编排]
  Step 1: 知识点 (T=0.3)         [可重试] ─┐
  Step 2: 概述 + 能力提升 (T=0.5) [可重试]  │ 每步独立落库
```

### 5.3 数据模型（核心表）

```sql
-- 学生表
CREATE TABLE students (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    age INTEGER,
    gender TEXT,
    grade TEXT,
    base_level TEXT,        -- 入门/初级/中级
    characteristics JSON,   -- ["内向", "喜欢挑战"]
    parent_contact TEXT,
    note TEXT,
    class_id INTEGER,
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    FOREIGN KEY (class_id) REFERENCES classes(id)
);

-- 班级表
CREATE TABLE classes (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    schedule_day TEXT,
    schedule_time TEXT,
    created_at TIMESTAMP
);

-- 课程记录表（一节课 = 一份报告）
CREATE TABLE course_records (
    id INTEGER PRIMARY KEY,
    student_id INTEGER NOT NULL,
    course_date DATE NOT NULL,
    project_folder TEXT,
    project_topic TEXT,
    knowledge_points JSON,
    ability_improvement TEXT,
    vocabulary JSON,
    content_summary TEXT,
    homework JSON,
    evaluation TEXT,
    screenshot_paths JSON,
    layout_template_id TEXT,
    report_docx_path TEXT,  -- 导出的 Word 文件路径
    report_pdf_path TEXT,
    status TEXT,            -- draft / finalized / archived
    ai_meta JSON,           -- AI 生成的元信息（用于"重新生成"复用上下文）
    last_auto_saved_at TIMESTAMP,
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    FOREIGN KEY (student_id) REFERENCES students(id)
);

-- 报告模板表
CREATE TABLE layout_templates (
    id TEXT PRIMARY KEY,    -- 用户可自定义 ID
    name TEXT,
    config JSON,            -- 完整模板配置
    is_builtin INTEGER,     -- 内置 vs 自定义
    created_at TIMESTAMP
);

-- 应用全局设置（Logo、默认主题、用户名等）
CREATE TABLE app_settings (
    key TEXT PRIMARY KEY,   -- 如 'global_logo' / 'default_template' / 'user_name'
    value JSON,             -- 灵活存储任意 JSON
    updated_at TIMESTAMP
);
-- 典型 key：
--   global_logo: { "path": "data/assets/logo.png", "uploaded_at": "..." }
--   default_template: "classic_default"
--   user_name: "李老师"
--   institution_name: "童趣编程工作室"
```

## 八、风险评估与应对

| 风险 | 等级 | 应对策略 |
|------|------|---------|
| AI 生成内容质量不稳定 | 🟡 中 | Prompt 多轮调优、老师可重试单条字段 |
| 4 页 A4 排版信息密度过载 | 🟡 中 | 优先做版面原型验证，必要时减少默认字号 |
| 老师上传截图不规范 | 🟢 低 | 提供裁剪/压缩工具，模板自动适配 |
| WeasyPrint 中文字体兼容 | 🟡 中 | 打包内置字体文件，避免依赖系统字体 |
| LangChain 版本升级导致 API 变化 | 🟢 低 | 用 Pydantic 抽象 + 自定义接入，弱化耦合 |
| API 限流/超时 | 🟢 低 | 重试机制 + 限流队列 + 友好错误提示 |
| pywebview 跨平台差异 | 🟡 中 | 优先保证 Windows/macOS，Linux 延后 |
| 数据丢失（学生档案/历史报告） | 🟡 中 | 启动时自动备份，导出 JSON 备份入口 |
| **草稿自动保存冲突** | 🟢 低 | 最后写入获胜 + 锁机制；用户编辑时显示"已自动保存"提示 |
| **Word 导入字段识别错位** | 🟡 中 | 导出 docx 用 Heading 1 明确分节；导入时 AI 辅助 + 用户确认 |
| **python-docx 中文字体回退** | 🟡 中 | 显式设置东亚字体（如"宋体"），不依赖系统默认 |
| **docx 编辑后格式丢失** | 🟢 低 | 提示用户避免在 Word 中删改 Heading 1 标题；提供"重置模板"入口 |
| **AI 单步重试导致上下文不一致** | 🟡 中 | 每次重试前快照旧值，支持"撤销重试"；UI 标记"AI 生成"vs"人工编辑" |
| **Logo 透明背景与白底冲突** | 🟢 低 | 上传时自动检测透明通道；白底模板预览时给 Logo 加浅色描边 |
| **Logo 比例失调（极扁/极高图片）** | 🟡 中 | 渲染时**始终保持原宽高比**，只缩放不裁剪；UI 提示"建议使用更接近方形的 Logo" |
| **Logo 替换后历史报告不一致** | 🟢 低 | Logo 作为"渲染时引用"而非"嵌入数据库"，历史 PDF 保留当时版本；新报告用新 Logo |

- ✅ 5 种典型 Python 教学项目（pygame/turtle/算法/小游戏/工具）均可正确解析
